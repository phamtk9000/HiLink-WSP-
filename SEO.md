# SEO — what's implemented, and what still limits this site

## Starting point
The site had effectively **zero SEO**: the page title was still Vite's default
(`hilink-workspace`), there was no meta description, no canonical, no Open
Graph tags, no `robots.txt`, no `sitemap.xml`, and no structured data. Good
news going in: all 84 images already had `alt` attributes and the heading
hierarchy was sound.

---

## 1 · Static head (`index.html`)
Read by **every** crawler, including the ones that don't run JavaScript.

- Real `<title>` and meta description targeting *premium coworking / serviced
  offices / Hanoi*
- `canonical`, `robots` (`index, follow, max-image-preview:large`), `author`,
  `theme-color`
- Full **Open Graph** set (`og:title/description/url/image/type/site_name`,
  `og:locale` = `en_US` + `vi_VN` alternate) and **Twitter** summary-large-image
- `preconnect` to the Google Fonts origins (removes a round-trip on first paint)
- `apple-touch-icon`
- **JSON-LD** `Organization` (NAP: address, phone, email, `areaServed`, `sameAs`)
  and `WebSite` — static, so always crawlable

## 2 · Per-route meta (`src/lib/seo.js`)
A dependency-free `useSeo()` hook (no react-helmet) that sets, per route:

- `<title>` (clamped to 65 chars, on a word boundary) and description (158)
- `canonical` matching the real URL
- Open Graph + Twitter, with the page's own image
- `robots: noindex` where appropriate
- `<html lang>` following the EN/VI toggle
- JSON-LD, tagged `data-seo="route"` so route changes only ever clear *their own*
  blocks and never touch the static Organization/WebSite schema

Wired into **13 hooks across 11 screens**: Home, Locations, LocationDetail,
Solutions index, SolutionPage, Forum, ArticlePage, About, Careers, Membership,
Partnerships, Legal, 404.

## 3 · Structured data (rich results)
| Schema | Where |
|---|---|
| `Organization`, `WebSite` | static, site-wide |
| `LocalBusiness` + `CoworkingSpace` (address, geo, phone) | every location page |
| `Service` | every solution page |
| `Article` (headline, image, datePublished, author, publisher) | every forum post |
| `BreadcrumbList` | locations, solutions, articles |

`LocalBusiness` + geo is the highest-value item here — it's what feeds Google
Maps / local pack results for "coworking Hoàn Kiếm" style searches.

## 4 · Crawl control
- **`sitemap.xml`** — 36 URLs, generated by `scripts/gen-seo.mjs` **from the
  actual data files**, so it can't drift from the routes. Runs automatically on
  every `npm run build` (or `npm run seo`).
- **`robots.txt`** — allows the site, disallows thin/duplicate tool states
  (`/recommend`, `/spaces`, `/portal`), points at the sitemap.
- Excluded from the sitemap on purpose: filter/tool states and legal pages.

## 5 · Bug fixed: soft 404s
`<Route path="*" element={<Homepage />} />` rendered **the homepage for every
unknown URL**. Every typo'd or spam URL returned a 200 with duplicate homepage
content — Google treats these as soft 404s and it dilutes the real homepage.
Replaced with a proper bilingual `NotFoundPage` marked `noindex, nofollow`.

## 6 · Social share image
`public/og-default.jpg` — 1200×630, generated from workspace photography with
the cream logo mark, contrast-checked (cream strokes at 246 against a ~28
background). 71 KB.

## Honest limitations — the things that still cap this site

### 1. It's a client-rendered SPA (biggest issue)
Google renders JavaScript, so it *will* see the per-route tags above. **Most
social crawlers do not** — Facebook, Zalo and LinkedIn read only the static
`index.html`. So a shared link to `/locations/obc` currently shows the generic
homepage preview, not that location's title/photo.

**Fix:** pre-render the routes at build time (SSG) or move to SSR/Next.js. With
a fixed route list and mostly static content, pre-rendering is the cheap win —
it would make every URL fully crawlable and give each page its own link preview.
(Couldn't be installed in this environment: pre-render plugins download a
headless Chromium, which the sandbox network blocks.)

### 2. EN and VI share one URL
The language toggle is React state — `/locations/obc` serves both languages at
the same address. Google can therefore only index **one** language version, and
`hreflang` is impossible (it requires distinct URLs per language). For a
Vietnamese-market site this is leaving real traffic on the table.

**Fix:** move language into the path (`/en/...`, `/vi/...`) or a subdomain, then
emit `hreflang` pairs. This is a routing refactor, not a tweak — worth planning
deliberately.

### 3. The 404 returns HTTP 200
The SPA server returns `index.html` for everything, so the new 404 page renders
but the *status code* is still 200. `noindex` handles the SEO risk; a true 404
status needs host config (on Vercel, a rewrite/status rule).

### 4. Things to confirm before launch
- **`SITE_URL`** in `src/lib/seo.js` and `scripts/gen-seo.mjs` is set to
  `https://hilink.vn` — change if the live domain differs (it's also read from
  `SITE_URL` env at build time).
- `sameAs` social URLs in `index.html` are placeholders — swap in the real
  LinkedIn/Facebook/Instagram profiles or delete them.
- Location `coords` are approximate — verify each in Google Maps.
- Submit the sitemap in Google Search Console after launch.

## Changelog — second audit pass (mobile PSI 67)

| Finding | Fix |
|---|---|
| 45 images, ~22 with `[Missed]` alt | Every content image (location galleries, unit photos, venue/membership/about/careers/partner images, homepage & enquiry photos) now has a descriptive, keyword-bearing alt built from context — location name, unit number, section, or "HiLink … Hanoi". Improves image-search visibility, not just the audit. `Title attribute [Missed]` is left as-is: the HTML `title` attribute is not an SEO factor and adding it to every image is noise |
| `404 page responded 200` (critical) | Static SPA hosting returns 200 for every path. Added a Vercel **Edge Middleware** (`middleware.js`): known routes pass through at ~0 cost and serve at 200; unknown URLs get a real HTTP **404** (plus `x-robots-tag: noindex`) while users still see the styled in-app 404 once React boots. The route list (`known-routes.json`) is generated at build time from the data files by `scripts/gen-seo.mjs`, so it can't drift — e.g. `/locations/obc` → 200, `/locations/nope` and `/wp-admin` → 404 |
| Small-text / hidden footer links (mailto + 3 socials) | Social links raised to 14px, normal case, 40px tap area; email/phone already 14.5px |
| Efficient cache lifetimes (images) | `/mid`, `/thumbs`, `/workspace-photos` etc. bumped from 7d to 30d + `stale-while-revalidate` |

### Requires a redeploy to take effect
`middleware.js`, `vercel.json` and the new headers only apply once pushed to Vercel. Also add `@vercel/edge` (already in `package.json`) — run `npm install` before deploying. **The audit was partly run against `hilink.vn` and partly against the Vercel preview; make sure the domain you re-audit is actually serving this build, or findings will look unfixed.**

### Still open — needs the routing refactor, not a tweak
The **hreflang** warnings ("incorrect language code", "no self-referencing annotation") come from one root cause: EN and VI share a single static `index.html`, so `/` and `/?lang=vi` return byte-identical HTML with the same canonical (`/`). A crawler can't confirm they're genuinely different-language pages, and the VI page canonicalising to EN contradicts the hreflang pair. The tags are valid and help JS-capable Google, but they can't be *fully* correct until each language has its own pre-rendered URL (`/en/…`, `/vi/…`) with its own canonical. That's the same SSG/path-based refactor already noted above, and it also fixes the thin-content / text-to-code ratio for non-JS crawlers on every route. Recommend scheduling it as a dedicated piece of work.
